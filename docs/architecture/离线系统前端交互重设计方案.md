# 离线系统前端交互重设计方案

> 状态：已按本文主要阶段落地到离线前后端。
>
> 本文同时作为后续验收清单。当前实现已覆盖统一 shell、三角色口径、JSON 任务包主流程、警员邀请卡、我的分片、参与记录、安全芯片管理、审计只读、分片视角移交、分片销毁入口和热备份下载；更完整的 A/B 双确认迁移与加密冷备份恢复仍可作为后续增强。

## 1. 背景和目标

当前离线桌面端功能基本能覆盖密钥生成、签名、安全芯片、通知、审计等能力，但交互结构比较像调试工具：

- 应用 shell 放在 `Dashboard.vue`，进入二级页面后失去左侧导航，只能靠悬浮“主页”按钮返回。
- 注册登录仍使用“用户名/邮箱”模型，不贴合警务场景。
- 角色包含 `coordinator`、`participant`、`guest` 等中间态，和测试大纲、在线系统的角色口径不一致。
- 安全芯片管理被拆成“导入安全芯片”，缺少统一列表和状态视图。
- 通知页以表格展示，警员需要点开详情才知道自己要参与哪个地址、哪个案件、什么操作。
- 密钥移交现在更像“密钥归属变更”，但真实语义应是警员 A 的某个私钥分片移交给警员 B。
- 在线系统导出的 JSON 任务包应该是主入口，手工输入只作为补充入口。

本次重设计的目标是：把离线端改成一个固定左右两栏布局的角色工作台，让管理员能发起和管理，警员能清楚处理自己的任务和分片，审计员能专注查看审计，整体交互与在线系统保持一致但不照搬在线系统的全部业务。

## 2. 设计原则

- 固定应用框架：登录/注册之外，所有页面都在统一 header + left nav + main content 内切换。
- 角色口径收敛：只保留管理员、警员、审计员三个角色。
- JSON 任务包优先：密钥生成和签名优先通过在线系统导出的 JSON 导入发起；手工录入只保留必要字段作为补充。
- 少填、可查、可选：能从服务端查到、从任务包解析出、从当前 SE 读取到、或能自动生成的字段，不要求用户手动填写。
- 警员工作台优先：警员不只“接受通知”，还要能看到自己参与过哪些地址、哪些分片、哪些 keygen/sign。
- 分片是管理对象：移交、销毁、参与记录都围绕“某个地址下某个警员持有的分片”表达。
- SE 统一管理：SE 不是个人资产，SE 列表、导入、状态检查放在同一个模块；分片和 `record_id` 才是隔离核心。
- 不过度设计案件：离线端可以展示任务包中的 `case_no/case_name`，但不承担完整案件管理。

## 3. 目标角色和权限

| 角色 | 内部值 | 能做什么 | 不能做什么 |
| --- | --- | --- | --- |
| 管理员 | `admin` | 用户管理、SE 管理、导入任务包、发起 keygen/sign、参与 keygen/sign、发起分片移交/销毁、备份恢复、查看审计 | 无 |
| 警员 | `officer` | 接受/拒绝 keygen/sign/transfer/destroy 邀请、执行本地 MPC/SE 操作、查看自己的分片和参与历史 | 发起 keygen/sign、管理用户、管理 SE、看全量审计 |
| 审计员 | `auditor` | 查看审计日志、查看只读任务/密钥摘要 | 发起操作、参与 MPC、管理用户和 SE |

需要同步调整：

- 服务端 `User.Role` 从 `admin/coordinator/participant/auditor/guest` 收敛为 `admin/officer/auditor`。
- 当前 `coordinator` 的发起能力并入 `admin`。
- 当前 `participant` 改名为 `officer`。
- 注册后的默认角色建议为 `officer`，管理员可以再调整为 `admin` 或 `auditor`。
- 用户管理页面只展示这三个角色，不再出现协调者、参与者、访客。

## 4. 应用框架

参考在线端 `online-client/frontend/src/App.vue` 的结构，把离线端 shell 上移到 `offline-client-wails/frontend/src/App.vue`：

```text
App.vue
├── public route: /login, /register
└── authenticated shell
    ├── Header: 系统名、当前用户、角色、WS 状态、客户端设置、退出
    ├── Aside: 按角色展示导航
    └── Main: router-view
```

改完后删除悬浮“主页”按钮。每个二级页面都保留左侧菜单，用户可以直接在安全芯片、账户、密钥生成、签名、审计之间切换。

页面视觉应保持业务工具风格：

- 不做营销式首页。
- 页面主体以表格、筛选、状态卡、步骤条为主。
- 管理员页面偏“列表 + 操作抽屉/弹窗”。
- 警员页面偏“待处理任务卡 + 我的分片 + 历史记录”。
- 审计页面偏“筛选 + 审计表格 + 详情抽屉”。

## 5. 导航信息架构

### 5.1 管理员导航

```text
工作台
离线任务
  - 导入任务包
  - 任务记录
密钥操作
  - 密钥生成
  - 交易签名
密钥与分片
  - 地址/密钥列表
  - 分片移交
  - 分片销毁

  - 导入安全芯片
账户管理
审计日志
备份恢复
客户端设置
```

说明：

- “密钥生成”和“交易签名”可以是独立路由，也可以合并在“离线任务”下用任务类型区分。建议保留独立入口，降低操作成本。
- “分片移交”和“分片销毁”属于密钥与分片模块，不应表达成账户所有权移交。
- “备份恢复”与在线系统保持口径，提供热备份、加密冷备份、恢复记录和校验。

### 5.2 警员导航

```text
我的工作台
待处理邀请
我的分片
参与记录
客户端设置
```

说明：

- 警员登录后默认看到待处理邀请和自己的分片摘要，不默认进入管理员式仪表盘。
- 待处理邀请要以醒目的任务卡展示，不以普通表格作为唯一入口。
- 我的分片展示“地址、案件编号、分片序号、门限、SE CPLC、record_id、状态、最近参与时间”。

### 5.3 审计员导航

```text
审计工作台
审计日志
任务记录
密钥摘要
客户端设置
```

说明：

- 审计员看到的是只读摘要和日志，不出现 keygen/sign 发起按钮。
- 审计筛选字段应与在线系统对齐：时间、用户、角色、动作、资源、案件编号、地址、结果。

## 6. 登录和注册

### 6.1 注册

注册界面保留简洁表单：

| 字段 | 说明 |
| --- | --- |
| 登录标识 | 必填。placeholder：`请输入手机号、警号或身份证号` |
| 昵称 | 可选。placeholder：`请输入姓名或常用昵称` |
| 密码 | 必填 |
| 确认密码 | 必填 |

去掉邮箱字段。后端可以保留数据库 `email` 字段作为历史字段，但新接口不应要求客户端传邮箱。

建议请求体：

```json
{
  "identifier": "P000001",
  "nickname": "张三",
  "password": "******"
}
```

如果为了减少后端改动，第一阶段也可以继续用 `username` 承载登录标识、增加 `nickname` 字段，但 UI 文案必须改成“手机号/警号/身份证号”。

### 6.2 登录

登录界面字段：

| 字段 | 说明 |
| --- | --- |
| 登录标识 | `手机号 / 警号 / 身份证号` |
| 密码 | 密码 |

连接设置不放在登录表单主体里。登录页右上角或卡片底部提供“服务器设置”按钮，点击后进入设置子页面或弹窗：

- 默认只展示“登录标识”和“密码”。
- 设置页优先只让用户填写服务器 IP 或域名，例如 `192.168.1.20`。
- 客户端根据 IP 自动生成：
  - HTTP 地址：`http://192.168.1.20:8080`
  - WebSocket 地址：`ws://192.168.1.20:8081/ws`
- 高级设置中再允许用户手动修改 HTTP 端口、WS 端口、协议和读卡器名称。
- 设置页提供“测试连接”按钮，分别检查 HTTP 登录端口和 WebSocket 端口是否可达。
- 本机默认值仍是 `127.0.0.1`，便于本机三窗口联调。

这样用户日常只需要知道“离线服务器 IP”，不用理解 HTTP/WS 两套地址。

### 6.3 输入框和选择器原则

离线端要尽量让操作通过点击、选择、扫描、导入完成，而不是让用户手写复杂字符串。

通用规则：

- 登录标识输入框要有明确提醒：`手机号 / 警号 / 身份证号`。
- 昵称字段可选，并提示“可填姓名或常用昵称，不影响登录”。
- 任务编号、案件编号、地址、消息哈希优先从 JSON 任务包解析。
- 参与警员用可搜索多选框，从服务端用户列表查询，不让管理员手写用户名。
- 地址用可搜索下拉或分片列表选择，不让管理员粘贴长地址作为主流程。
- SE CPLC 通过“读取当前 SE”获取，不让用户手写。
- `record_id`、session_key、room、临时文件名全部系统生成，不暴露给普通操作员填写。
- 手工录入页面保留，但放在“高级/手工创建”入口，并给每个字段写清楚示例和用途。
- 任何需要用户命名的地方都提供自动建议，例如任务备注、SEID、备份文件名。

建议的自动命名规则：

```text
SEID: SE-YYYYMMDD-001
任务备注: 案件号 + 操作类型 + 日期
备份文件: offline-backup-YYYYMMDD-HHMMSS
手工任务编号: MANUAL-KEYGEN-YYYYMMDD-HHMMSS
```

## 7. 管理员核心流程

keygen/sign 的发起和结果交付都应围绕 JSON 包闭环设计。具体包格式以
[在线/离线数据包与 MPC 边界设计](../../offline-server/docs/online_offline_mpc_contract.md)
为准，前端只负责让这个闭环好用：

```text
在线系统下载 offline_task_<task_no>.json
    -> 离线系统导入任务包
    -> 管理员选择参与警员并发起 keygen/sign
    -> 警员确认并执行 MPC/SE
    -> 离线系统生成 offline_result_<task_no>.json
    -> 在线系统上传结果包
```

页面上允许展示地址、公钥、签名等结果摘要，但“下载 JSON 结果包”必须是 keygen/sign 完成后的主操作。不要让用户复制页面字段再回在线系统手工粘贴。

落地时，“密钥生成”和“交易签名”二级页面只保留为清晰入口和说明页，主操作统一跳转到“离线任务”页面完成：

- 导入 `offline_task_<task_no>.json`。
- 解析并展示任务摘要。
- 选择管理员/警员参与方。
- 通过 WebSocket 发起 keygen/sign。
- 下载 `offline_result_<task_no>.json`。

这样可以避免出现一套 JSON 包流程、一套手工表单流程并行，减少后续对接在线系统时的歧义。

### 7.1 安全芯片管理

合并当前“导入安全芯片”为“安全芯片管理”页面：

```text
安全芯片管理
├── 顶部操作：读取当前 SE、导入当前 SE、刷新列表
├── 筛选：SEID、CPLC、状态
└── 列表：SEID、CPLC、状态、登记人、登记时间、最近使用时间、备注
```

导入流程：

1. 管理员插入 SE。
2. 点击“读取当前 SE”。
3. 桌面端读取 CPLC。
4. 系统自动建议 SEID，例如 `SE-20260528-001`。
5. 管理员只在需要时补充备注。
6. 点击“导入安全芯片”。

注意：SE 不归属个人，只作为可用硬件资源统一管理。分片归属体现在 `KeyShard.username` 和 `record_id`。

### 7.2 离线任务导入

管理员发起 keygen/sign 的主入口应是 JSON 文件：

```text
导入任务包
├── 拖拽/选择 JSON 文件
├── 自动解析任务类型、任务编号、案件编号、地址/交易哈希、推荐门限
├── 展示解析摘要
└── 下一步：选择参与警员并发起
```

任务包解析后进入具体流程：

- `custody_keygen`：进入密钥生成流程。
- `sign`：进入交易签名流程。
- `transfer`：进入分片移交流程。
- `destroy`：进入分片销毁流程。

手工录入入口保留在每个流程右上角“手工创建”，字段只保留核心信息，避免和在线系统的案件录入重复。

导入任务包后，页面应保存并展示：

- `task_no`
- `task_type`
- `payload_hash`
- `case_no`，如果任务包提供
- `transaction_no`，仅签名任务
- `from_address/message_hash/display`，仅签名任务
- `threshold_policy`，仅 keygen 或任务包提供门限策略时

如果任务包校验失败，应直接阻止发起，并展示失败原因，例如包类型不匹配、`payload_hash` 不一致、任务编号重复。

### 7.3 密钥生成

管理员流程：

1. 导入托管钱包生成任务包，或手工输入任务编号/案件编号/门限参数。
2. 从可搜索警员列表中勾选参与者，管理员本人也可以被选为参与者。
3. 系统展示 2-of-3 等门限摘要、参与者、预计使用 SE。
4. 管理员发起邀请。
5. 警员在各自桌面端确认。
6. 服务端下发 GG20 参数，桌面端执行 MPC/SE。
7. 完成后展示地址、公钥、分片状态，并提供“下载 keygen 结果包”。

页面应显示状态步骤：

```text
任务已导入 -> 邀请已发送 -> 参与方确认 -> MPC 执行中 -> 结果已生成 -> 结果包已导出
```

下载的结果包必须是契约中的 `custody_keygen_result`，文件名建议为：

```text
offline_result_<task_no>.json
```

页面摘要只显示：

- 任务编号。
- 案件编号。
- 托管地址。
- 公钥。
- 门限策略。
- 离线引用号。
- `payload_hash/result_hash`。

结果包不得包含参与方用户名、CPLC、`encrypted_shard`、GG20 share 明文或密文。

### 7.4 交易签名

管理员流程：

1. 导入在线系统导出的签名任务包。
2. 系统根据地址加载该地址的可用分片持有人。
3. 管理员从可用分片持有人中勾选满足门限的警员组合。
4. 发起签名邀请。
5. 警员确认并执行。
6. 管理员查看签名摘要，并下载签名结果包。

页面重点展示：

- 案件编号，可选。
- 地址。
- 消息哈希。
- 门限。
- 可参与警员。
- 当前签名组合。
- 签名结果。

下载的结果包必须是契约中的 `sign_result`，文件名建议为：

```text
offline_result_<task_no>.json
```

页面摘要只显示：

- 任务编号。
- 交易编号。
- 案件编号。
- 签名地址。
- 消息哈希。
- 签名值。
- 签名格式。
- 离线引用号。
- `payload_hash/result_hash`。

离线端不负责广播交易。在线系统导入 `sign_result` 后，再校验签名恢复地址并广播。

### 7.5 分片移交

移交的对象是“某个警员持有的某个私钥分片”，不是账户所有权。

管理员流程：

1. 在密钥/分片列表中选择地址。
2. 展开分片列表，选择警员 A 的分片。
3. 点击“移交分片”。
4. 从警员列表中选择接收警员 B，填写或选择移交原因。
5. 系统给 A 和 B 发送确认邀请。
6. A 确认移出，B 确认接收。
7. 服务端更新分片持有人，必要时生成新的 `record_id` 或记录 transfer version。
8. 审计日志记录完整链路。

建议邀请卡文案：

```text
分片移交确认
地址：0x1234...abcd
案件编号：CASE-2026-001
分片：2 / 3
发起人：admin
移出警员：A
接收警员：B
原因：人员离职交接
```

关键约束：

- 只有管理员可发起。
- A 和 B 都必须确认。
- 移交不改变地址、不改变案件、不改变其他警员分片。
- 移交后的“我的分片”应从 A 转到 B。

### 7.6 分片销毁

销毁也应围绕分片表达：

- 管理员可对某个地址发起整体销毁流程。
- 系统实际向每个分片持有人发送销毁邀请。
- 警员确认后，本地 SE 删除对应 `record_id`。
- 所有必要分片完成后，地址/离线密钥状态置为 destroyed。

## 8. 警员核心体验

### 8.1 我的工作台

警员首页不再是空泛欢迎页，而是一个可操作的工作台：

```text
顶部状态
├── WebSocket 连接状态
├── 当前读卡器
├── 当前 SE CPLC
└── 待处理数量

待处理任务卡
├── Keygen 邀请
├── Sign 邀请
├── 分片移交邀请
└── 销毁确认

我的分片
├── 地址
├── 案件编号
├── 分片序号
├── 门限
├── 状态
└── 最近参与时间

参与历史
├── 密钥生成
├── 交易签名
├── 移交
└── 销毁
```

### 8.2 待处理邀请

邀请不应只在通知表格里出现，应作为醒目的卡片：

```text
[待确认] 交易签名邀请
案件编号：CASE-2026-001
地址：0x1234...abcd
消息哈希：0xabcd...
发起人：admin
需要操作：插入指定 SE 后确认参与

[读取 SE 并同意] [拒绝]
```

交互要求：

- 卡片上直接展示核心信息，不要求先点“详情”。
- “同意”按钮应先读取当前 SE CPLC，并展示读取失败原因。
- 任务执行中显示进度和当前阶段。
- 成功后显示“结果已回传”。
- 失败后显示错误，并支持复制错误详情。

### 8.3 我的分片

警员需要知道自己参与了哪些地址：

| 字段 | 说明 |
| --- | --- |
| 地址 | 托管钱包或签名地址 |
| 案件编号 | 来自在线任务包，可为空 |
| 分片编号 | `shard_index` |
| 门限 | `required_signers / total_parties` |
| SE CPLC | 实际写入或读取的 SE |
| record_id | SE 内部记录编号 |
| 状态 | active/transferred/destroyed |
| 最近操作 | keygen/sign/transfer/destroy |

这需要服务端补一个“当前用户分片列表”接口，或者在现有密钥查询接口上增加按用户过滤。

### 8.4 参与历史

警员参与历史不等于审计日志全量。建议提供自己的简明记录：

- 时间。
- 类型：密钥生成、签名、移交、销毁。
- 地址。
- 案件编号。
- 结果：成功、失败、拒绝。
- 错误摘要。

## 9. 审计和备份

### 9.1 审计

离线端审计与在线系统保持风格一致：

- 顶部筛选：时间范围、用户、角色、动作、资源、案件编号、地址、结果。
- 表格字段：时间、用户、角色、动作、资源、案件编号、地址、结果、错误。
- 详情抽屉展示请求摘要、会话编号、任务编号、参与方、结果摘要。

审计员只能查看，不能发起或确认任何敏感操作。

### 9.2 冷热备份

离线端也应提供“备份恢复”模块，和在线系统口径保持一致：

- 热备份：本地 SQLite 数据库快照。
- 加密冷备份：导出加密备份文件，用于离线保管。
- 校验：校验备份文件哈希和元数据。
- 恢复记录：记录谁在何时恢复了哪个备份。

备份模块只对管理员可见。

## 10. 数据和接口调整清单

### 10.1 用户

建议模型：

```text
identifier: 登录标识，手机号/警号/身份证号，唯一
nickname: 昵称/姓名，可选
password_hash
role: admin/officer/auditor
status: active/disabled
```

第一阶段可以沿用 `username` 字段承载 `identifier`，但接口和 UI 文案改为登录标识。

需要改的地方：

- 注册请求去掉必填 `email`。
- 登录错误文案从“用户名或密码错误”改为“登录标识或密码错误”。
- 用户列表去掉邮箱列，增加昵称列。
- 用户角色只允许 `admin/officer/auditor`。
- 可参与 keygen/sign 的用户来自 `admin + officer`，审计员不可参与。

### 10.2 安全芯片

列表接口需要返回：

```text
se_id
cplc
status
registered_by
created_at
last_used_at
remark
```

SE 不绑定个人归属；如果保留 `owner` 字段，应改名为 `remark` 或 `location_hint`，避免误导。

### 10.3 分片

分片应成为 UI 重点对象：

```text
offline_key_id
address
case_no
task_no
shard_index
username
record_id
se_cplc
status
created_at
updated_at
last_used_at
```

需要补充：

- 管理员按地址/案件/用户查询分片。
- 警员查询自己的分片。
- 移交接口以 `shard_id` 或 `offline_key_id + shard_index + from_user` 定位，不以 `logical_owner` 定位。

### 10.4 任务和通知

通知消息需要带足够展示字段，避免用户点开 JSON：

```json
{
  "type": "sign_invite",
  "session_key": "...",
  "task_no": "...",
  "case_no": "CASE-2026-001",
  "address": "0x...",
  "message_hash": "0x...",
  "initiator": "admin",
  "party_index": 2,
  "required_signers": 2,
  "total_parties": 3,
  "summary": "交易签名邀请"
}
```

keygen、transfer、destroy 也应按同样原则补齐摘要字段。

## 11. 页面改造优先级

### 阶段一：先把壳和角色理顺

- 把离线端 shell 上移到 `App.vue`。
- 删除悬浮主页按钮。
- 导航按 `admin/officer/auditor` 分组展示。
- 注册/登录表单改为登录标识 + 昵称，去掉邮箱。
- 登录页增加“服务器设置”按钮，设置页默认只填服务器 IP，自动生成 HTTP/WS 地址。
- 用户管理只保留管理员、警员、审计员。

这是最先做的，因为它决定用户是否能顺畅切换页面。

### 阶段二：把警员体验改好

- 新建或重做“我的工作台”。
- 把通知表格改成待处理任务卡 + 历史列表。
- 我的分片页面展示警员持有地址和分片状态。
- 邀请卡直接展示地址、案件、门限、发起人和按钮。

这是最影响实际联调体验的一层。

### 阶段三：整理管理员业务入口

- 安全芯片管理合并列表和导入。
- 离线任务导入改成 JSON 主入口。
- 密钥生成/签名页面增加任务包摘要、步骤条和结果包下载。
- 密钥管理改成地址/分片视角。

### 阶段四：分片移交和备份恢复

- 移交从“密钥归属”改成“警员 A 的某个分片移交给警员 B”。
- A/B 双确认通知。
- 备份恢复页面按在线系统风格落地。
- 审计详情补齐移交、销毁、备份恢复记录。

## 12. 验收标准

- 登录后任意页面都有左侧导航，不需要回主页再进入下一页。
- 注册界面无邮箱字段，登录标识文案明确为手机号/警号/身份证号。
- 登录页不直接暴露复杂连接表单；用户可通过“服务器设置”只填写 IP，并一键测试 HTTP/WS 连接。
- 用户管理只出现管理员、警员、审计员。
- 管理员可以发起 keygen/sign，也可以作为参与方被选中。
- 管理员选择警员、地址、安全芯片、分片时优先通过搜索、下拉、列表完成，不要求手写复杂编号。
- 警员登录后第一眼能看到待处理邀请、自己的分片和参与记录。
- 警员无需点开 JSON 就能知道邀请对应的地址、案件编号、操作类型和发起人。
- 安全芯片列表和导入在同一个模块。
- 分片移交明确展示 A、B、地址、分片、原因，并要求双方确认。
- 审计员看不到发起和确认操作入口。
- 任务包导入是 keygen/sign 的主流程入口，手工录入是补充入口。
- keygen 完成后可以下载符合契约的 `custody_keygen_result` JSON 结果包。
- sign 完成后可以下载符合契约的 `sign_result` JSON 结果包。
- 页面展示的地址、公钥、签名只能作为摘要，不能替代 JSON 结果包导出。

5. 备份恢复第一阶段是否只做数据库热备份和加密冷备份下载，恢复操作是否先保留为管理员手动确认？
