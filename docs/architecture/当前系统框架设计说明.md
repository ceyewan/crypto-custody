# 虚拟货币存管提控原型系统框架设计说明

## 1. 设计目标

虚拟货币存管提控原型系统面向缴获虚拟货币账户的安全存储、分案管理、提取控制和交易执行场景。系统采用在线与离线分离的总体架构：在线系统负责业务管理、账户管理、交易申请、链上交互和审计留痕；离线系统负责密钥生成、门限签名和安全芯片存储，避免敏感密钥材料暴露在联网环境中。

该架构的核心目标包括：

- 支持缴获虚拟货币账户信息的联网可靠存储和查询。
- 支持以案件为单位管理账户、交易和提取流程。
- 支持离线环境下的密钥安全存储和门限签名。
- 支持在线端发起交易、离线端签名、在线端广播的提取控制流程。
- 支持关键操作日志记录，为审计和测试提供依据。
- 支持本地测试链或测试网络环境下的余额查询和交易验证。

## 2. 总体架构

系统由在线服务端、在线客户端、离线服务端、离线客户端和安全芯片 Applet 五部分组成。

```text
在线访问终端
    |
    v
在线客户端 Web
    |
    v
在线服务端
    |        \
    |         \ 区块链 RPC
    |          v
    |       本地测试链 / 测试网络
    |
    | 任务包 / 结果包
    v
离线客户端
    |
    v
离线服务端
    |
    v
安全芯片
```

在线系统和离线系统之间不做实时数据同步。在线系统作为案件、账户、交易和审计的主数据源；离线系统作为密钥生成、密钥安全存储和签名执行环境。两者通过离线任务包和结果包交换必要信息。

## 3. 在线系统设计

在线系统由在线服务端和在线客户端组成，负责联网环境下的业务管理和链上交互。

### 3.1 在线服务端

在线服务端提供 HTTP API，主要职责包括：

- 用户管理：用户注册、登录、登出、密码修改、角色管理。
- 权限控制：区分管理员、操作员等不同角色的操作权限。
- 账户管理：导入缴获账户、批量导入账户、查询账户、查询链上余额。
- 分案管理：以案件编号为核心组织账户和交易信息。
- 交易管理：创建提取交易申请、生成待签名交易信息、接收签名结果、广播交易、查询交易状态。
- 链上交互：通过可配置 RPC 连接本地测试链或测试网络，完成余额查询、nonce 获取、gas 估算、交易广播和交易回执查询。
- 审计留痕：记录用户、账户、案件、交易和系统管理相关关键操作。
- 备份恢复：对在线端关键业务数据进行热备份、冷备份和恢复验证。

在线服务端保存的是业务数据、账户地址、案件信息、交易记录和审计日志，不保存明文私钥。

### 3.2 在线客户端

在线客户端是面向业务人员的 Web 操作界面，主要页面包括：

- 用户登录与个人信息页面。
- 用户管理页面。
- 账户管理页面。
- 分案管理页面。
- 交易管理页面。
- 备份恢复与审计查询页面。

在线客户端用于完成联网侧的日常操作，例如导入账户、绑定案件、创建提取交易、查看交易状态和查询审计记录。

## 4. 离线系统设计

离线系统由离线服务端、离线客户端和安全芯片组成，部署在隔离环境中，负责密钥和签名相关能力。

### 4.1 离线服务端

离线服务端负责协调多方参与的密钥生成和签名流程，主要职责包括：

- 离线用户登录和角色管理。
- 密钥生成会话创建与协调。
- 签名会话创建与协调。
- WebSocket 实时消息分发。
- 安全芯片登记信息管理。
- MPC manager 后台进程管理。

离线服务端不承担案件主数据管理，只处理当前离线任务所需的任务编号、账户地址、消息哈希、参与方和安全芯片信息。

### 4.2 离线客户端

离线客户端是离线操作终端，负责和操作员、安全芯片、离线服务端交互，主要能力包括：

- 接收在线系统导出的离线任务包。
- 展示任务包中的案件编号、账户地址、交易哈希和操作说明。
- 参与门限密钥生成。
- 参与门限签名。
- 调用本地能力访问安全芯片。
- 输出签名结果包，供在线系统导入。

### 4.3 安全芯片

安全芯片用于保存密钥相关敏感材料。系统在密钥生成过程中将密钥材料加密后存储，并将必要的解密材料或密钥索引写入安全芯片。读取和删除操作需要经过授权校验，避免未授权访问。

安全芯片承担以下职责：

- 保存密钥相关敏感材料。
- 支持按用户名和账户地址定位密钥记录。
- 支持授权读取。
- 支持授权删除。
- 支持 CPLC 等芯片标识信息读取。

通过安全芯片存储，系统避免密钥材料以明文形式保存在普通文件或联网服务端中，从而满足离线私钥安全存储的设计要求。

## 5. 在线与离线的数据边界

系统明确区分在线主数据和离线敏感数据。

### 5.1 在线系统保存的数据

- 用户信息和角色信息。
- 案件信息。
- 缴获账户地址、币种、余额和案件归属关系。
- 交易申请、审批、状态和链上交易哈希。
- 审计日志。
- 备份文件和恢复记录。

### 5.2 离线系统保存或处理的数据

- 密钥生成会话信息。
- 签名会话信息。
- 门限签名参与方信息。
- 加密后的密钥材料。
- 安全芯片中的密钥相关敏感材料。
- 当前任务包中的案件编号、账户地址和交易哈希。

### 5.3 不进行实时同步的原因

在线系统和离线系统之间不做实时同步，原因包括：

- 离线系统需要保持隔离，不能依赖在线网络连接。
- 案件信息会持续变更，在线系统更适合作为主数据源。
- 离线系统只需要处理当前任务，不适合保存长期案件主数据。
- 通过任务包和结果包交互，可以保证每次离线操作都基于在线系统导出的最新业务状态。

## 6. 任务包与结果包机制

在线系统和离线系统通过任务包、结果包完成数据交换。

### 6.1 任务包

任务包由在线系统生成，用于把当前操作所需信息带入离线环境。任务包可包含：

```json
{
  "taskId": "TASK-2025-001",
  "caseNo": "CASE-2025-002",
  "operation": "sign",
  "fromAddress": "0x...",
  "toAddress": "0x...",
  "amount": "0.01 ETH",
  "messageHash": "0x...",
  "reason": "案件资金提取控制测试",
  "createdBy": "operator",
  "createdAt": "2025-05-08T10:00:00Z"
}
```

离线系统读取任务包后，只基于该任务执行密钥生成或签名，不修改案件主数据。

### 6.2 结果包

结果包由离线系统生成，用于把离线执行结果带回在线系统。结果包可包含：

```json
{
  "taskId": "TASK-2025-001",
  "caseNo": "CASE-2025-002",
  "address": "0x...",
  "messageHash": "0x...",
  "signature": "0x...",
  "participants": ["user1", "user2"],
  "securityChipIds": ["SE001", "SE002"],
  "completedAt": "2025-05-08T10:05:00Z"
}
```

在线系统导入结果包后，完成签名验证、交易广播、状态更新和审计记录。

## 7. 主要业务流程

### 7.1 账户导入与分案管理

1. 管理员或操作员登录在线系统。
2. 创建案件，填写案件编号和案件描述。
3. 导入缴获账户地址，填写币种、账户说明和案件编号。
4. 在线系统保存账户信息，并可通过区块链 RPC 查询账户余额。
5. 后续可按案件编号查询账户列表和相关交易记录。

### 7.2 密钥生成与安全存储

1. 在线系统根据业务需要生成密钥生成任务包。
2. 操作员将任务包带入离线环境。
3. 离线客户端读取任务包并发起门限密钥生成流程。
4. 多个离线参与方共同完成密钥生成。
5. 密钥相关敏感材料加密存储，并写入安全芯片。
6. 离线系统生成结果包，返回地址、公钥信息、安全芯片标识和任务结果。
7. 在线系统导入结果包，将生成地址与案件或账户记录关联。

### 7.3 提取控制与交易广播

1. 操作员在在线系统中选择案件和账户，创建提取交易申请。
2. 在线系统检查账户余额、交易金额和权限。
3. 审批通过后，在线系统构建待签名交易并生成消息哈希。
4. 在线系统导出签名任务包。
5. 离线系统读取任务包，由满足门限要求的参与方完成签名。
6. 离线系统生成签名结果包。
7. 在线系统导入签名结果包，验证签名有效性。
8. 验证通过后，在线系统广播交易到本地测试链或测试网络。
9. 在线系统查询交易回执，更新交易状态并记录审计日志。

### 7.4 备份与恢复

1. 管理员在在线系统中触发热备份或冷备份。
2. 热备份用于保存在线数据库快照。
3. 冷备份用于导出加密备份文件并保存到离线介质。
4. 测试恢复时，可模拟删除案件或账户数据。
5. 管理员通过备份文件恢复数据。
6. 恢复后通过在线系统查询案件、账户和交易记录，验证数据一致性。

## 9. 测试链设计

测试环境建议使用 Ganache 搭建本地以太坊测试链，在线服务端通过可配置 RPC 地址连接本地链。

建议配置项包括：

```text
ETH_RPC_URL=http://127.0.0.1:7545
ETH_CHAIN_ID=5777
ETH_CONFIRM_TIMEOUT=60s
ETH_GAS_LIMIT=21000
```

采用本地测试链的原因：

- 测试账户和余额可控。
- 出块速度快，交易确认稳定。
- 不依赖外部网络和测试币水龙头。
- 便于重复执行余额查询、交易广播和回执检查。

## 10. 设计原则总结

系统采用“在线管业务、离线管密钥”的设计原则：

- 在线系统是案件、账户、交易和审计的主数据源。
- 离线系统是密钥生成、门限签名和安全芯片存储的执行环境。
- 两边不做实时同步，通过任务包和结果包进行受控交换。
- 在线端不保存明文私钥，离线端不维护长期案件主数据。
- 测试链可配置，便于在 Ganache、本地测试链或测试网络之间切换。

该设计既满足在线业务管理和链上交易测试需要，也保留了离线密钥安全存储和门限签名能力，能够支撑测试大纲中对存管提控原型系统的主要测试要求。
