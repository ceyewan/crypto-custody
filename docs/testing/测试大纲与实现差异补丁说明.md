# 测试大纲与实现差异补丁说明

本文档不修改《虚拟货币存管提控原型系统测试大纲》，只记录当前实现与测试大纲之间需要澄清或修正执行口径的地方，便于后续验收时保持测试大纲、设计文档和代码实现一致。

## 1. 安全芯片管理边界

测试大纲中包含“安全芯片管理模块功能、安全”测试项。结合在线/离线隔离架构，该测试项不应由在线系统实现。

建议验收口径：

- 在线系统不提供安全芯片导入、芯片列表、警员分片映射、私钥分片查询接口。
- 安全芯片登记、CPLC 读取、芯片状态、私钥存储、私钥查询、私钥移交、私钥销毁均由离线系统完成。
- 在线系统只记录案件、账户、托管钱包地址、交易和离线任务/结果引用。

## 2. 私钥移交和销毁边界

测试大纲要求测试私钥移交和私钥销毁。该能力涉及安全芯片、警员参与方和密钥材料，应在离线系统完成。

建议验收口径：

- 离线系统执行私钥移交和销毁。
- 在线系统不保存私钥、不保存分片、不执行移交和销毁。
- 如需在线留痕，在线系统最多接收离线系统导出的结果摘要或证明哈希。

## 3. 国密 SM2 与当前 GG20/Ethereum 实现口径

测试大纲中明确提到“基于国密 SM2 算法的门限签名”。当前代码实现主要围绕 `GG20 + Ethereum` 地址和签名流程。

这是当前实现与测试大纲之间最大的潜在差异。

建议后续二选一明确：

- 若验收必须严格使用 SM2，则需要替换或补充当前 MPC 二进制与验签逻辑，使离线端输出 SM2 兼容签名。
- 若验收接受原型系统以现有 GG20/Ethereum 流程验证门限签名机制，则需要在验收材料中明确“当前原型以以太坊签名流程作为演示实现”。

## 4. 本地接口测试运行条件

当前 `online-server/tests/account_test`、`online-server/tests/user_test`、`online-server/tests/transaction_test` 中有多组用例需要访问 `localhost:8080` 上运行的在线服务。

在受限沙箱中运行 `go test ./...` 时，本地 TCP 连接可能被禁止，导致测试失败。这不是业务代码编译错误。

建议执行口径：

1. 先启动在线服务。
2. 准备 `.env`、默认管理员密码、测试链 RPC 和必要测试签名。
3. 在允许访问 `localhost:8080` 的环境中运行接口测试。

## 5. 旧测试中不存在用户 Token 的预期

当前 `online-server/tests/user_test.go` 中存在一个旧用例，它期望使用不存在用户生成的 token 访问当前用户接口时仍返回成功。当前中间件会尝试根据 token 中用户名加载用户模型，用户不存在时返回未授权。

建议修正测试执行口径：

- 不存在用户对应的 token 应视为无效登录态。
- 当前接口返回 `401` 更符合认证安全边界。

该点属于旧测试与认证安全逻辑不一致，不建议为通过旧测试而放宽认证。

## 6. 备份恢复深度

测试大纲要求冷热备份与恢复。当前在线端后端已提供热备份、冷备份文件记录、下载、校验和恢复记录更新。

需要进一步补齐的验收能力：

- 冷备份文件实际加密。
- 从备份文件真实恢复数据库内容。
- 恢复后自动校验案件、账户、交易数据一致性。

当前实现已建立接口和数据模型，后续仍需增强恢复执行逻辑。

## 7. 提取控制广播条件

在线端负责构建交易、生成待签名哈希、导出签名任务、导入离线签名和广播。广播依赖同一服务进程中保存的待签名交易缓存，或后续持久化完整 unsigned payload 后重建交易。

当前后端已将 `TransactionManager` 调整为单例，以支持同一服务进程内的构建、签名和广播闭环。

后续建议：

- 将 unsigned payload 持久化到数据库。
- 服务重启后也能根据数据库记录重建待广播交易。

## 8. 结论

当前代码实现应按以下边界验收：

- 在线系统覆盖用户、案件、账户、交易、审计、备份、测试数据和离线任务交互。
- 离线系统覆盖安全芯片、私钥、门限密钥生成、门限签名、移交和销毁。
- 若测试大纲要求严格 SM2，需要补充算法实现；否则需明确当前 GG20/Ethereum 原型口径。
