# 虚拟货币存管提控原型系统离线端全量设计方案

> 文档状态：已归档。
>
> 本文是早期离线端全量设计草案，保留用于追溯设计演进。当前离线系统的权威详细设计请以
> [offline_system_detailed_design.md](../../offline-server/docs/offline_system_detailed_design.md) 为准。
>
> 主要差异：新版设计已经补齐测试大纲映射、在线/离线 JSON 包边界、SE 统一管理、`record_id`
> 稳定记录键、会话级 GG20 manager、`parties/signing_index` 规则、私钥查询/移交/销毁和最小可验收版本。

## 1. 文档目标

本文档描述虚拟货币存管提控原型系统离线端的目标设计。离线端是系统的密钥和签名安全域，负责安全芯片、门限密钥生成、门限签名、私钥存储、私钥查询、私钥移交、私钥销毁，以及与在线系统之间的离线任务包和结果包交互。

离线端设计以覆盖测试大纲为边界，不承担在线案件管理、在线账户批量管理、在线交易广播和在线数据库备份职责。

## 2. 离线端职责边界

### 2.1 离线系统负责

- 离线用户管理：管理员、协调员、参与者等离线角色。
- 安全芯片管理：芯片登记、CPLC 读取、芯片状态检查。
- 私钥安全存储：将收缴私钥或密钥材料在离线环境中写入安全芯片或安全存储体系。
- 私钥信息查询：授权后查询已存储私钥或密钥材料的受控结果。
- 私钥移交：在离线环境中完成密钥控制权转移。
- 私钥销毁：在离线环境中完成密钥材料删除或销毁。
- 门限密钥生成：为案件托管钱包生成门限密钥和地址。
- 门限签名：对在线系统导入的交易哈希执行门限签名。
- 安全芯片和警员参与方映射：记录哪个离线参与者、哪个安全芯片参与了某个密钥或签名流程。
- 离线任务包导入：读取在线系统导出的托管钱包生成任务和签名任务。
- 离线结果包导出：输出托管钱包地址、签名结果、操作证明等结果。
- 离线审计日志：记录密钥、芯片、签名、移交、销毁等敏感操作。

### 2.2 离线系统不负责

- 在线案件主数据管理。
- 在线账户列表和 1000 条账户批量导入。
- 链上 nonce、gas、余额查询和交易广播。
- 在线交易状态查询和回执确认。
- 在线系统冷热备份恢复。

离线系统只处理在线系统导出的任务数据，不直接连接区块链网络。

## 3. 总体架构

离线端由三部分组成：

```text
离线操作终端
    |
    v
离线客户端 Wails 桌面应用
    |
    +--> 本地 MPC 二进制
    |
    +--> 安全芯片 / 读卡器 / JavaCard Applet
    |
    +--> 离线任务包 / 结果包文件
    |
    v
离线协作服务端
    |
    +--> HTTP API
    |
    +--> WebSocket 协调服务
    |
    +--> MPC manager 进程
    |
    +--> SQLite 本地数据库
```

离线客户端用于警员和协调员操作。离线服务端用于多参与方会话协调、用户认证、WebSocket 消息分发和会话记录。安全芯片用于保存密钥相关敏感材料或密钥解密材料。

## 4. 跨平台设计

离线客户端需要支持 Windows 和 macOS，必要时支持 Linux。跨平台核心策略如下：

- 离线客户端使用 Wails 构建桌面应用。
- 前端使用 Vue，后端使用 Go。
- MPC 核心能力封装为平台相关本地二进制。
- 应用启动或执行 MPC 时，根据 `runtime.GOOS` 和 `runtime.GOARCH` 选择对应二进制。
- 二进制随客户端打包或嵌入，离线环境不依赖在线下载。
- 任务包和结果包使用 JSON 文件，避免平台差异。
- 安全芯片驱动、读卡器名称、Applet AID 等通过本地配置管理。

MPC 二进制命名约定：

```text
gg20_keygen_windows_amd64.exe
gg20_signing_windows_amd64.exe
gg20_keygen_darwin_arm64
gg20_signing_darwin_arm64
gg20_keygen_linux_amd64
gg20_signing_linux_amd64
```

跨平台注意事项：

- Windows 下需要处理 `.exe`、读卡器驱动和安装包权限。
- macOS 下需要处理可执行权限、应用签名、沙盒权限和读卡器访问。
- Linux 下需要处理 PC/SC 服务、可执行权限和动态库依赖。
- 临时密钥文件应在流程结束后删除。
- 平台差异不得影响任务包和结果包格式。

## 5. 角色权限设计

| 角色 | 定位 | 权限 |
| --- | --- | --- |
| admin | 离线管理员 | 用户管理、安全芯片登记、系统配置、审计查询、私钥销毁确认 |
| coordinator | 协调员 | 发起密钥生成会话、发起签名会话、导入任务包、导出结果包 |
| participant | 参与者 | 接受密钥生成或签名邀请，插入安全芯片并执行本地操作 |
| auditor | 审计员 | 查询离线审计日志和操作证明 |
| guest | 游客 | 基础登录态或无敏感权限 |

权限要求：

- 密钥生成和签名会话只能由 admin 或 coordinator 发起。
- 参与者只能处理分配给自己的会话。
- 安全芯片登记和私钥销毁需要 admin 权限。
- 私钥查询、移交、销毁必须记录审计日志。

## 7. 任务包和结果包

### 7.1 托管钱包生成任务包

在线系统导出，离线系统导入。

```json
{
  "taskNo": "TASK-2025-0001",
  "taskType": "custody_keygen",
  "caseNo": "CASE-2025-001",
  "coinType": "ETH",
  "thresholdPolicy": "2_of_3",
  "createdBy": "online-operator",
  "createdAt": "2025-05-08T10:00:00Z"
}
```

离线系统导入后，由协调员选择参与者和安全芯片，执行门限密钥生成。

### 7.2 托管钱包生成结果包

离线系统导出，在线系统导入。

```json
{
  "taskNo": "TASK-2025-0001",
  "taskType": "custody_keygen_result",
  "caseNo": "CASE-2025-001",
  "coinType": "ETH",
  "custodyAddress": "0x...",
  "publicKey": "0x...",
  "offlineRefNo": "OFFLINE-KEY-2025-0001",
  "completedAt": "2025-05-08T10:10:00Z"
}
```

结果包不包含参与者名单、安全芯片编号和分片内容。在线系统只需要地址和任务结果。

### 7.3 签名任务包

在线系统导出，离线系统导入。

```json
{
  "taskNo": "TASK-2025-0002",
  "taskType": "sign",
  "caseNo": "CASE-2025-001",
  "transactionNo": "TX-2025-0001",
  "fromAddress": "0x...",
  "toAddress": "0x...",
  "value": "0.01",
  "coinType": "ETH",
  "messageHash": "0x...",
  "reason": "案件资产提取控制测试"
}
```

离线系统只对 `messageHash` 签名，不负责查询链上 nonce、gas 或余额。

### 7.4 签名结果包

离线系统导出，在线系统导入。

```json
{
  "taskNo": "TASK-2025-0002",
  "taskType": "sign_result",
  "transactionNo": "TX-2025-0001",
  "messageHash": "0x...",
  "signature": "0x...",
  "completedAt": "2025-05-08T10:20:00Z"
}
```

## 8. 核心业务流程

### 8.1 案件托管钱包生成

1. 在线系统导出托管钱包生成任务包。
2. 离线协调员导入任务包。
3. 离线协调员选择门限参数、参与者和安全芯片。
4. 离线服务端创建密钥生成会话。
5. 参与者登录离线客户端并插入安全芯片。
6. 各参与者接受邀请。
7. 离线客户端调用本地 MPC keygen 二进制。
8. 密钥材料加密后写入安全芯片或安全存储。
9. 离线服务端记录生成地址和会话状态。
10. 离线系统导出托管钱包生成结果包。

### 8.2 签名流程

1. 在线系统导出签名任务包，包含待签名哈希。
2. 离线协调员导入任务包。
3. 离线协调员发起签名会话。
4. 参与者登录离线客户端并插入安全芯片。
5. 参与者接受签名邀请。
6. 离线客户端从安全芯片读取授权数据。
7. 离线客户端调用本地 MPC signing 二进制。
8. 离线服务端收集签名结果。
9. 离线系统导出签名结果包。
10. 在线系统导入签名结果并广播交易。

### 8.3 收缴私钥离线导入

1. 操作员在离线环境录入或导入收缴私钥材料。
2. 离线系统校验私钥和地址一致性。
3. 离线系统将私钥或派生密钥材料加密写入安全芯片。
4. 离线系统记录密钥引用和审计日志。
5. 如需要在线留痕，离线系统导出操作结果摘要，不导出私钥。

### 8.4 私钥移交

1. 离线管理员发起移交流程。
2. 系统校验发起人权限和目标参与者状态。
3. 原持有人和目标持有人按要求完成授权。
4. 离线系统执行密钥材料转移、重加密或重分发。
5. 旧持有人权限失效。
6. 记录审计日志和移交证明。

### 8.5 私钥销毁

1. 离线管理员发起销毁流程。
2. 系统校验权限和销毁对象。
3. 安全芯片或本地安全存储执行删除。
4. 系统验证删除后不可读取。
5. 记录销毁证明和审计日志。

## 9. 门限签名执行方式

离线端默认使用本地 MPC 二进制执行门限密钥生成和签名：

- `gg20_keygen` 用于门限密钥生成。
- `gg20_signing` 用于门限签名。
- `gg20_sm_manager` 用于协调 MPC 多方通信。

执行原则：

- 在线系统负责生成交易哈希。
- 离线系统只签名哈希。
- 每个参与者只在本机处理自己的分片。
- 临时明文文件流程结束后必须删除。
- 最终只导出地址或签名结果，不导出分片。

## 10. 安全设计

- 离线系统部署在物理隔离或受控网络环境中。
- 私钥、分片和安全芯片访问结果不得进入在线系统。
- 日志中不得记录私钥、助记词、keystore、分片明文。
- 读取和删除安全芯片数据必须要求授权签名或等效授权机制。
- 任务包和结果包应保存哈希，用于完整性验证。
- 安全芯片丢失、禁用或销毁应在离线系统中记录。
- 数据库和任务文件应做本地备份。

## 12. 当前实现对应性审核

### 12.1 已经对应的部分

- `offline-server` 已有用户模型和角色：`admin`、`coordinator`、`participant`、`guest`。
- `offline-server` 已有密钥生成会话和签名会话模型。
- `offline-server` 已有 WebSocket 消息类型，覆盖密钥生成邀请、响应、参数、结果和签名邀请、响应、参数、结果。
- `offline-server` 启动时会启动 `gg20_sm_manager` 进程。
- `offline-server` 已有安全芯片记录模型 `Se`，包含 `se_id` 和 `cpic`。
- `offline-client-wails` 已有 Wails 桌面应用框架。
- `offline-client-wails` 前端已有登录、用户、密钥生成、签名、安全芯片导入、通知和测试页面。
- `offline-client-wails` 已有本地 MPC 服务，支持 keygen 和 signing。
- `offline-client-wails` 已经按 `runtime.GOOS` 和 `runtime.GOARCH` 选择嵌入式 MPC 二进制，覆盖 Windows、macOS 和 Linux 的主要二进制文件。
- `offline-client-wails` 已有安全芯片服务，支持存储、读取、删除和 CPLC 获取。
- 当前实现已经有本地测试入口，便于验证 MPC 和安全芯片基础能力。

### 12.2 尚未完整对应的部分

- 当前离线端还没有标准化的“在线任务包导入”和“离线结果包导出”文件流程。
- 当前离线端模型中有 `Case`，但离线系统目标态不应承担完整案件主数据管理，建议弱化为任务上下文或兼容保留。
- 当前离线端还缺少统一的 `OfflineTask` 模型，用于记录任务包哈希、结果包哈希、导入导出状态。
- 当前离线端还缺少独立审计日志模型，现有运行日志不能替代安全审计。
- 当前安全芯片模型较简单，缺少 owner、status、registered_by 等管理字段。
- 当前私钥移交流程没有形成明确接口和页面。
- 当前私钥销毁已有底层删除能力，但缺少完整销毁流程、权限确认、销毁证明和审计记录。
- 当前私钥信息查询已有读取能力，但缺少测试大纲要求的授权查询流程和禁止批量查询等控制说明。
- 当前性能基准测试没有形成固定入口，尚不能直接输出标准签名和门限签名的性能对比报告。
- 当前系统文档和部分前端文字仍使用“云端服务器”表述，建议统一为“离线协作服务端”或“离线服务端”。

### 12.3 后续建议

第一阶段优先补齐：

1. 任务包导入和结果包导出。
2. 离线审计日志。
3. 安全芯片状态字段和芯片管理页面完善。
4. 私钥查询、移交、销毁流程的页面和接口。
5. 性能基准测试入口。

第二阶段再优化：

1. 离线任务包和结果包签名或哈希校验。
2. 离线数据库备份恢复。
3. 更完整的跨平台打包和驱动检查。
4. 国密 SM2 门限算法与当前 GG20/Ethereum 流程之间的兼容说明或替换方案。
