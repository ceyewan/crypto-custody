# 安全芯片设计文档

## 1. 定位

安全芯片是离线系统的本地硬件安全边界，用于保存每个参与者访问密钥材料所需的短秘密值，并对读取、删除操作做授权校验。

当前实现中，安全芯片不直接保存完整私钥，也不直接执行门限签名。门限 keygen/sign 由桌面端调用 GG20 二进制完成；安全芯片用于保护加密 local share 的 32 字节解密密钥。

## 2. 当前 Applet 能力

Applet 路径：

```text
offline-client/secured/src/securitychip/SecurityChipApplet.java
```

支持三类 APDU 指令：

| 指令 | INS | 说明 |
| --- | --- | --- |
| STORE_DATA | `0x10` | 写入或覆盖一条记录 |
| READ_DATA | `0x20` | 读取一条记录，需要 ECDSA 授权签名 |
| DELETE_DATA | `0x30` | 删除一条记录，需要 ECDSA 授权签名 |

每条记录结构：

| 字段 | 长度 | 说明 |
| --- | --- | --- |
| userName | 32 字节 | 桌面端传入 `sha256(username)` |
| address | 20 字节 | 托管钱包以太坊地址 |
| message | 32 字节 | 用于解密 local share 的随机密钥 |

当前容量：

```text
MAX_RECORDS = 100
```

## 3. 数据保护模型

keygen 完成后，桌面端会得到当前参与者自己的 local share。处理方式：

1. 生成随机 32 字节 `randomKey`。
2. 使用 `randomKey` 加密并压缩 local share。
3. 将加密后的 local share 返回离线服务端保存。
4. 将 `randomKey` 写入安全芯片。
5. 删除本地明文 local share 临时文件。

sign 时：

1. 离线服务端把加密 local share 下发给对应参与者。
2. 服务端生成安全芯片读取授权签名。
3. 桌面端用授权签名从安全芯片读取 `randomKey`。
4. 桌面端解密 local share。
5. 桌面端调用 `gg20_signing`。
6. 签名结束后删除临时明文文件。

因此服务端只有加密 share，安全芯片只有解密 key。任意一边单独泄露，都不足以还原 local share。

## 4. 授权校验

READ 和 DELETE 需要 ECDSA 签名。

签名对象：

```text
sha256(username) || address
```

其中：

- `sha256(username)` 为 32 字节。
- `address` 为 20 字节二进制地址。
- 总长度 52 字节。

Applet 内置 ECDSA 公钥，使用 NIST P-256 / secp256r1 曲线校验 DER 格式签名。对应私钥由离线服务端或受控离线授权模块持有，用于生成读取/删除授权。

注意：授权签名只能证明“允许读取/删除某用户某地址的记录”，不能替代完整的业务审批。业务审批仍由离线服务端角色、会话和审计控制。

## 5. 与离线服务端的关系

离线服务端记录安全芯片的管理信息：

| 字段 | 说明 |
| --- | --- |
| se_id | 芯片业务编号 |
| cplc | 芯片唯一标识 |
| owner | 当前持有人 |
| status | active/lost/disabled/destroyed |
| registered_by | 登记人 |

离线服务端不直接读取芯片内容。读取动作发生在桌面客户端连接读卡器时。

服务端负责：

- 登记芯片 CPLC。
- 校验参与者上报的 CPLC 是否匹配分配芯片。
- 记录芯片状态。
- 生成读取/删除授权签名。
- 审计所有读取、删除、移交、销毁流程。

桌面端负责：

- 连接读卡器。
- 选择 Applet。
- 发送 APDU。
- 读取 CPLC。
- 写入、读取、删除安全芯片记录。

## 6. 芯片状态

| 状态 | 说明 |
| --- | --- |
| active | 可用于 keygen/sign |
| lost | 已挂失，不允许新会话使用 |
| disabled | 管理停用，不允许新会话使用 |
| destroyed | 已销毁或已清除，不允许使用 |

安全芯片丢失时，离线系统应：

1. 将芯片状态置为 `lost`。
2. 阻止该芯片参与新的 keygen/sign。
3. 记录丢失审计。
4. 对相关托管钱包发起迁移流程。
5. 迁移完成后，在线系统只需要批量更新安全钱包地址。

## 7. 私钥移交和销毁

私钥移交本质上是 local share 控制权变化，应在离线系统内完成：

1. 管理员发起移交。
2. 原持有人和目标持有人完成授权。
3. 目标持有人完成新芯片写入。
4. 旧芯片记录删除或失效。
5. 离线服务端更新 KeyShard 持有人和 CPLC。
6. 写审计日志。

私钥销毁流程：

1. 管理员发起销毁。
2. 系统校验对象、权限和审批。
3. 桌面端对芯片执行 DELETE。
4. 再执行 READ 验证不可读取。
5. 服务端将 KeyShard 状态置为 `destroyed`。
6. 生成销毁证明和审计日志。

## 8. 安全要求

- 日志不得记录 `randomKey`、local share 明文、私钥、助记词。
- READ/DELETE 必须要求授权签名。
- 授权签名应绑定用户名和地址，避免跨用户/跨地址复用。
- 客户端临时明文文件必须在流程结束后删除。
- 芯片 CPLC 必须与服务端登记记录匹配。
- 芯片丢失、禁用、销毁必须阻止后续会话使用。
- 测试环境生成的 ECDSA 授权密钥不得用于生产。

## 9. 当前实现差距

已经具备：

- JavaCard Applet 支持存储、读取、删除。
- READ/DELETE 有 ECDSA 签名校验。
- 桌面端 `SecurityService` 支持 Store/Read/Delete/GetCPLC。
- 记录键使用服务端下发的 32 字节 hex `record_id` 和以太坊地址。
- 存储值为 32 字节，符合当前 `randomKey` 设计。
- WebSocket 销毁流程会下发 DELETE 授权，桌面端删除后立即 READ 验证不可读取，再由服务端标记分片和密钥为 `destroyed`。

仍需补齐：

- 芯片状态字段和状态流转页面。
- 芯片丢失、禁用、销毁后的业务拦截。
- 移交流程中的新旧芯片双向确认。
- 可导出的销毁证明文件。
- 授权签名私钥的生产级保管方案。
- Applet 公钥轮换和版本管理。
- 批量测试时的芯片模拟或测试模式说明。
